# Spec 백로그 10개 — grill 세션 결정 기록 (2026-07-16)

> 목적: Codex가 spec 문서 ~10세트를 작성하기 전에, 사용자·Claude grill 세션에서 확정한
> 결정(스캐폴딩)을 기록한다. Codex는 이 문서를 입력으로 각 spec의
> requirements / acceptance-criteria / test-scenarios / api-spec / ui-spec / traceability를 작성한다.
> 상태: **완성** — D1~D5 = 사용자 확정. D6~D8 = Claude 추천 기본값 채택
> (사용자 미심문 — spec PR 리뷰가 최종 체크포인트, 뒤집으면 해당 spec만 재작성).

## 배경

- Codex 토큰 여유가 더 많아, 이번 라운드는 **spec 문서 작성 = Codex** 가 담당한다
  (평시 규칙 "spec 작성 = Claude"의 1회성 재배치. 구현·리뷰·머지 절차는 기존과 동일).
- 완성된 spec은 `pending_specs` 큐(index.yml)에 순서대로 쌓여 무인 배치가 소진한다.
- 현재 active spec `2026-06-20-saramin-jd-image-ocr`는 구현 완료(#93 머지) 상태의 stale 포인터
  → 첫 spec PR에서 archive로 이동하고 새 큐로 교체한다.

## 확정 결정

### D1. 백로그 범위 = 로드맵 가로지르기 (A③ → B → C)

ATS 하나로는 10개가 안 나온다. Phase A③(ATS) → Phase B(결과 보존·이력서 관리) →
Phase C(계정·수익화)를 가로질러 10개를 구성한다. EC2 실배포(ADR-004)도 큐에 포함(위치는 D8).

### D2. 템플릿·키워드 사전 = 백로그 제외 + 사이드바에서 제거

- 두 항목은 6/14 SaaS 셸 목업에서 온 장식 메뉴로, **기능 정의가 어디에도 없음**.
- 정의 없는 항목을 백로그에 넣지 않는다. 대신 "지키지 않을 약속"을 UI에서 제거한다
  (#88 "맞춤 첨삭" 제거와 동일 패턴).
- 제거 작업은 **#2 spec의 ui-spec에 편입** — #2가 어차피 `lockedItems`(AppShell.tsx)를
  처음 수정하는 spec이므로(분석 내역 해금) 같은 곳에서 외과적으로 처리.

### D3. 백로그 구성·순서 확정 (10개, 큐 순서 = 표 순서)

| # | spec | 요지 |
|---|------|------|
| 1 | ATS 점수·포맷 진단 | 마지막 "준비중" 분석 옵션 해금. 기존 preview 패턴(신규 엔드포인트 또는 응답 확장) |
| 2 | 분석 내역 (localStorage) | 경량 보존 먼저. + 사이드바 정리(분석 내역 해금, 템플릿·키워드 사전 제거) |
| 3 | DB 영속화 기반 | PostgreSQL + Flyway + compose 서비스 추가 (D4) |
| 4 | 분석 내역 서버 저장 | #2를 #3 위로 이관. localStorage는 비로그인 폴백으로 재사용 |
| 5 | 이력서 관리 | 다중 이력서 저장·선택. 사이드바 해금 |
| 6 | EC2 실배포 | ADR-004 분리 컨테이너 + 도메인·HTTPS. #7 전 배치가 핵심 (D8) |
| 7 | 계정·인증 | 구글 OAuth 단독 (D5). 가짜 프로필(김현준) → 실물 |
| 8 | 내역·이력서 계정 연동 | 익명 데이터 → 계정 마이그레이션 |
| 9 | 요금제·사용량 제한 | 횟수 제한 모델 (D6). 요금제 메뉴 해금 |
| 10 | 결제 연동 | 토스페이먼츠 빌링 (D7) |

- 2단계 접근(#2 localStorage → #4 DB 이관)은 로드맵 문구를 따른 의도적 결정:
  spec을 밤 단위로 작게 유지, #3이 blocked(ADR-006)돼도 #2까지는 유저 가치 존속.
  #4 spec은 localStorage 코드를 버리지 않고 "비로그인 폴백 + 서버 동기화"로 재사용하도록 쓸 것.
- roadmap.md 개정을 동반한다 (아래 "로드맵 개정").

### D4. DB 엔진 = PostgreSQL 〔사용자 확정〕

- 근거 ①: 저장 대상 대부분이 AI 응답 JSON(점수·strengths·gaps·키워드 배열) → **JSONB**로
  통째 저장 + 인덱싱. 정규화 스키마 최소 유지.
- 근거 ②: 향후 vectorstore 필요 시 **pgvector** 확장으로 같은 DB에서 임베딩 검색 해결
  (별도 벡터 DB 운영 불필요) — 사용자 결정 근거.
- 마이그레이션 = **Flyway** (Claude 추천 기본값. Spring Boot 표준, SQL 파일 기반이라 리뷰 용이).
- compose에 `postgres` 서비스 추가 → #3부터 ADR-006 필수 의존성 차단 대상.
- #3 spec 작성 시 이 결정을 ADR로 승격해 decisions.md에 기록할 것.

### D5. 인증 = 구글 OAuth 단독 〔사용자 확정〕

- 비밀번호를 아예 받지 않는다 (해싱·재설정·이메일 발송 인프라 전부 불필요).
- 근거: 타겟(취준·이직 개발자)의 구글 계정 보유율, "비밀 키는 서버에서만" 보안 규칙의 연장,
  이메일 발송이라는 외부 의존(ADR-006 차단 대상) 제거.
- 세션 = **서버 세션 쿠키** (Claude 추천 기본값. 단일 백엔드라 JWT 이점 없음).
- 이메일+비밀번호 가입은 필요해지면 추후 별도 spec으로 추가(역방향보다 쉬움).

### D6. 요금제 = 횟수 제한(미터링) 〔Claude 추천 기본값〕

- 무료 = 전 기능 × 월 N회 분석(N은 #9 spec에서 확정), 유료 = 무제한 또는 큰 한도.
- 근거 ①: 이 서비스의 변동비는 Gemini 호출뿐 → "횟수"가 원가와 1:1 대응하는 유일한 축.
- 근거 ②: 무료에서 전 기능을 체험해야 유료 전환 동기가 생기는 제품 성격.
- 근거 ③: 구현 최단 — 계정별 사용량 카운터 + 월 리셋. #7 계정 테이블 위에 바로 얹힘.
- 기능 게이팅(무료 기능 잠금)은 "잠금 메뉴 0개" 방향(D2·로드맵)과 모순이라 배제.

### D7. 결제 = 토스페이먼츠 빌링(정기결제) 〔Claude 추천 기본값〕

- 근거 ①: 국내 서비스·원화 결제 — Stripe는 국내 가맹 지원이 제한적이라 배제.
- 근거 ②: 포트원(구 아임포트)은 멀티 PG 애그리게이터 — 단일 PG로 시작하는 MVP엔
  추상화 계층이 과함. PG 교체 필요가 생기면 그때 도입 검토.
- 근거 ③: 토스페이먼츠는 개발자 문서·테스트 환경(샌드박스 키)이 국내 PG 중 가장 정돈되어
  있어 무인 구현·테스트 루프(Codex)에 유리. 구독 = 빌링키 방식.
- High-risk(결제·외부 API): 자동 머지 금지, 사람 에스컬레이션 대상 (기존 루프 규칙).

### D8. EC2 배포 = 큐 6번, #7(계정) 앞 〔Claude 추천 기본값〕

- "병렬 트랙"이지만 직렬 큐에서는 위치가 필요하다. **#5(이력서 관리)와 #7(계정) 사이**에 둔다.
- 근거 ①: #7 구글 OAuth는 **고정 도메인 + HTTPS redirect URI가 전제** — 배포가 먼저 있어야
  운영 리다이렉트 설정이 가능. #10 결제도 HTTPS 필수.
- 근거 ②: #3~#5에서 DB가 생긴 직후 배포해야 운영 DB·Flyway 마이그레이션 경로가 조기 검증됨.
- 근거 ③: #1~#5는 로컬 검증으로 충분한 반면, #7부터는 운영 환경 없이는 반쪽 검증.
- CD·배포 워크플로 구현 = Codex 담당(기존 역할 분리). High-risk(배포·CI) — 자동 머지 금지.

- `pending_specs` 순서 = 로드맵 순서로 일치시킨다.

## 큐 삽입 정책 (운영 중 새 요구 발생 시)

- 외형·색·버튼 등 결정/계약 없는 변경 → 큐 무시, Codex에 변경요청 직접(배치와 병렬).
- 계약·아키텍처·문서화된 결정 뒤집기 → 새 spec을 `pending_specs` 원하는 위치에 splice
  (spec PR = 사용자 머지). High-risk는 자동 머지 제외.
- **기반 결정(DB 엔진 등 다수 spec이 전제)은 중간 삽입 금지** — 이번 grill처럼 선결한다.

## 용어 (glossary 후보)

- **ATS** (Applicant Tracking System): 기업의 이력서 자동 파싱·필터링 시스템.
  JDSnack의 "ATS 분석" = 파싱 안전성(표·이미지·비표준 섹션 감지) + 구조 진단(표준 섹션
  유무·순서) + JD 대비 키워드 최적화("기계가 찾는가" 관점).
- **pending_specs**: index.yml의 순서 큐. 무인 배치가 밤새 소진하는 spec 백로그 (#96).
- **큐 전진**: 구현 PR에 active spec archive 이동 + 첫 pending 승격을 원자적으로 포함 (doc-lifecycle.md).
- **잠금 메뉴**: AppShell.tsx `lockedItems` — UI가 노출하는 미구현 약속. 현재 5개.
