# 분석 실행 전 통합 검증 게이트 계획

## Summary

분석(진단/매칭) 실행 **전에** 이력서·JD·분석 항목을 한 번에 검증하고, 통과 전에는 분석 시작 버튼을 비활성화해 `/api/diagnose(/file)`·`/api/match/preview` 호출과 파일 업로드가 일어나지 않게 한다. 프론트·백엔드가 동일 기준(이력서 50/10,000, JD 50/10,000)을 쓰고, 백엔드는 방어적으로 재검증한다. 구현은 Codex가 담당.

## 변경 범위

- 문서: 새 active spec 추가, 직전 `2026-06-14-frontend-result-ux` spec archive 이동, `.agent-os/standards/index.yml`·`AGENTS.md` 활성 spec 포인터 갱신(이 문서 PR).
- 구현(프론트): `frontend/src/App.tsx`의 게이트(`canStart`)를 통합 검증 결과로 대체, JD 길이 검증을 게이트 단계로 끌어올림. 기존 `validateJdText`/`validateResumeText` 재사용.
- 구현(백엔드): 신규 로직 없음. `DiagnoseService`/`MatchPreviewService`의 기존 재검증을 계약으로 고정하고 테스트로 보장. 파일 추출 본문 최소 길이 재검증 확인.

## 구현 지침

- 게이트: 파일 모드 이력서는 "지원 형식 파일 선택됨", 텍스트 모드는 `validateResumeText` 통과, JD는 `validateJdText` 통과, 분석 항목 1개 이상. 모두 통과 시에만 버튼 활성.
- 사유 표시: 기존 검증 메시지를 인라인으로 노출. 비활성 버튼은 `aria-disabled`.
- 임계값은 새로 정의하지 않고 기존 함수/상수를 재사용해 프론트·백엔드 일치 유지.
- 검증: 프론트 `npm run lint`/`npm test`/`npm run build`/`npm run test:e2e`, 백엔드 `./gradlew test`. e2e에 "짧은 JD에서 네트워크 호출 미발생" 케이스 추가, 백엔드에 방어적 재검증 테스트 추가.

## 제외 범위

- 임계값(50/10,000) 변경, 새 API/스키마, 새 ErrorCode.
- 이력서 텍스트 직접 입력 UI 신설.
- 결과 화면·셸·결과 표시 컴포넌트 변경.

## 컨테이너 운영 기준

- 로컬 개발/검증은 `compose.local.yaml`(또는 `npm run dev` + 백엔드 로컬) 기준. 배포 영향 없음.

## 열린 질문 / 결정 필요

- (결정 필요) JD 최소 50자 임계값이 적절한지: JD는 보통 길지만, 링크 수집 본문이 짧게 추출되는 경우가 있어 50자가 과하게 느슨/엄격한지 확인 필요. 이번 spec은 현행 값을 유지하되, 조정 시 별도 spec.
- (결정 필요) 파일 모드에서 "지원 형식 파일 선택됨"만으로 게이트를 통과시키면, 추출 본문이 짧을 때 사용자는 업로드 후 백엔드 400을 받는다. 이 경험을 허용할지(백엔드 방어로 충분) vs. 업로드 전 클라이언트 추정 경고를 둘지.
- (결정 필요) 인라인 사유를 항목별로 모두 나열할지, 가장 우선순위 높은 1건만 보여줄지(현행 `formError`는 단일 메시지).
- (확인 완료) 프론트 텍스트 모드 이력서 입력은 **존재하지 않음** — 이력서는 파일 업로드 전용(PDF/DOCX, ≤10MB; `frontend/src/App.tsx` 섹션 2, `resumeFile` 단일 경로). 따라서 게이트의 이력서 조건은 "지원 형식 파일 선택됨"으로 확정하고, 이력서 본문 최소 길이(50자)는 프론트 게이트 대상이 아니라 추출 후 백엔드 방어 검증(`DiagnoseService` `TEXT_TOO_SHORT`)으로만 다룬다. requirements.md의 "텍스트 입력 모드" 분기와 AC-01의 텍스트 모드 판정은 현행 UI에 해당 사항 없음(구현 시 무시 가능).
