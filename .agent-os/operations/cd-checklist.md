# CD 체크리스트

## 목적

CD는 배포 자동화 자체보다 **배포 가능한 상태인지 검증하는 절차**입니다.

1차 MVP에서는 실제 배포 파이프라인을 만들기 전에 배포 조건, 순서, 확인 항목을 문서로 고정합니다.

## 현재 단계

- 상태: 컨테이너 빌드 검증 v1
- `.github/workflows/container.yml`에서 `main` 반영 후 백엔드 컨테이너 빌드와 compose 스모크 테스트를 검증합니다.
- 1차 MVP에는 외부 AI 비밀값 배포 검증이 없습니다.

## 기본 배포 모델

- `frontend`, `backend`를 분리 컨테이너로 배포합니다.
- `frontend/Dockerfile`, `backend/Dockerfile`를 각각 빌드합니다.
- reverse proxy 또는 ingress 뒤에서 하나의 서비스처럼 노출합니다.
- AWS EC2, ECS 또는 유사한 컨테이너 환경을 배포 대상으로 둡니다.

## 배포 전 조건

- PR 체크리스트 완료
- 머지 규칙 통과
- CI 체크리스트 통과
- 컨테이너 빌드 검증 통과
- 릴리즈 체크리스트 확인
- `GET /api/health` 경로가 구현되어 있음
- 1차 MVP API 계약이 `api-spec.md`와 일치함

## 배포 절차 초안

1. 프론트 컨테이너 이미지를 빌드합니다.
2. 백엔드 컨테이너 이미지를 빌드합니다.
3. 두 컨테이너를 같은 네트워크에 배치합니다.
4. reverse proxy 또는 ingress 라우팅을 적용합니다.
5. `/api/health`를 확인합니다.
6. 핵심 진단 흐름을 수동 검증합니다.

## 배포 후 확인

- `/api/health`가 정상 응답합니다.
- 메인 화면이 로드됩니다.
- 정상 길이 이력서 입력 시 준비중 안내가 표시됩니다.
- 빈 입력, 짧은 입력, 긴 입력 오류가 사용자 친화적으로 표시됩니다.
- 서버 로그에 민감정보가 출력되지 않습니다.

## 롤백 기준

- 애플리케이션 기동 실패
- 헬스체크 실패
- 컨테이너 빌드 또는 기동 실패
- 핵심 화면 미노출
- 진단 요청이 문서 계약과 다르게 응답

롤백 방식:

- 직전 정상 프론트/백엔드 이미지 조합으로 되돌립니다.
- 배포 실패 원인을 `incident-playbook.md` 기준으로 기록합니다.
- 재발 방지 항목을 `ci-checklist.md` 또는 `release-checklist.md`에 반영합니다.

## 2차 MVP 추가 조건

외부 AI 연동이 시작되면 아래 항목을 추가합니다.

- 서버 비밀값 주입 확인
- 외부 AI 타임아웃과 재시도 정책 확인
- 민감정보 로그 출력 여부 확인
- `Security Reviewer` 검토 완료
