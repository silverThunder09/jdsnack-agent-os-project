# CD 체크리스트

## 목적

CD는 배포 자동화 자체보다 **배포 가능한 상태인지 검증하는 절차**입니다.

1차 MVP에서는 실제 배포 파이프라인을 만들기 전에 배포 조건, 순서, 확인 항목을 문서로 고정합니다.

## 현재 단계

- 상태: 컨테이너 빌드 검증 v1
- `.github/workflows/container.yml`에서 `main` 반영 후 백엔드 컨테이너 빌드와 `/api/health`를 검증합니다.
- 1차 MVP에는 외부 AI 비밀값 배포 검증이 없습니다.

## 기본 배포 모델

- React 정적 빌드 결과물을 Spring Boot 정적 리소스로 포함합니다.
- Spring Boot 단일 JAR를 빌드합니다.
- `backend/Dockerfile`로 컨테이너 이미지를 빌드합니다.
- AWS EC2, Elastic Beanstalk, ECS 중 하나로 배포 대상을 확정합니다.

## 배포 전 조건

- PR 체크리스트 완료
- 머지 규칙 통과
- CI 체크리스트 통과
- 컨테이너 빌드 검증 통과
- 릴리즈 체크리스트 확인
- `GET /api/health` 경로가 구현되어 있음
- 1차 MVP API 계약이 `api-spec.md`와 일치함

## 배포 절차 초안

1. 프론트엔드 프로덕션 빌드를 생성합니다.
2. 빌드 결과물을 백엔드 정적 리소스 경로에 포함합니다.
3. 백엔드 단일 JAR를 빌드합니다.
4. Docker 이미지를 빌드합니다.
5. 컨테이너를 실행합니다.
6. `/api/health`를 확인합니다.
7. 핵심 진단 흐름을 수동 검증합니다.

## 배포 후 확인

- `/api/health`가 정상 응답합니다.
- 메인 화면이 로드됩니다.
- 정상 길이 이력서 입력 시 준비중 안내가 표시됩니다.
- 빈 입력, 짧은 입력, 긴 입력 오류가 사용자 친화적으로 표시됩니다.
- 서버 로그에 민감정보가 출력되지 않습니다.

## 롤백 기준

- 애플리케이션 기동 실패
- 헬스체크 실패
- 컨테이너 빌드 또는 기동 실패
- 핵심 화면 미노출
- 진단 요청이 문서 계약과 다르게 응답

롤백 방식:

- 직전 정상 JAR로 되돌립니다.
- 배포 실패 원인을 `incident-playbook.md` 기준으로 기록합니다.
- 재발 방지 항목을 `ci-checklist.md` 또는 `release-checklist.md`에 반영합니다.

## 2차 MVP 추가 조건

외부 AI 연동이 시작되면 아래 항목을 추가합니다.

- 서버 비밀값 주입 확인
- 외부 AI 타임아웃과 재시도 정책 확인
- 민감정보 로그 출력 여부 확인
- `Security Reviewer` 검토 완료
