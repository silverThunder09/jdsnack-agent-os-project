# 주요 결정 기록

## ADR-001 단일 저장소 운영

- 결정: 문서, 백엔드, 프론트엔드를 한 저장소에서 관리합니다.
- 이유: 구조 파악이 쉽고 학습/개발 흐름을 한 번에 관리할 수 있습니다.

## ADR-002 문서 우선 개발

- 결정: 기능 구현 전 spec과 standards를 먼저 정리합니다.
- 이유: 구현과 문서가 어긋나면 유지보수가 어려워집니다.

## ADR-003 MVP는 단순 흐름 우선

- 결정: 복잡한 계정보다 이력서/JD 분석 핵심 흐름을 먼저 완성합니다.
- 이유: 가장 먼저 검증해야 할 가치는 분석 품질과 사용 흐름입니다.

## ADR-004 최종 배포는 분리 컨테이너형

- 결정: 프론트엔드와 백엔드는 각각 독립 컨테이너로 배포합니다.
- 이유: 역할 분리가 명확하고, 프론트 정적 자산과 백엔드 API를 독립적으로 교체하기 쉽습니다.
- 이유: 1.5차 MVP와 2차 MVP에서 업로드, 외부 AI 연동, 보안 설정이 늘어나도 배포 경계를 유지하기 좋습니다.
- 운영 기준: 로컬과 CI는 `compose.yaml`을 기준으로 통합 검증하고, 운영 환경에서도 reverse proxy 또는 ingress 뒤에서 `frontend`와 `backend`를 분리 유지합니다.
