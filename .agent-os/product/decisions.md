# 주요 결정 기록

> **번호 체계(2026-07-18 일원화)**: ADR·DECISION·`.agent-os/adr/` 파일은 **하나의 연속 번호**를 공유합니다.
> 이 파일의 ADR-001~006·DECISION-007~013에 이어 기술 ADR 파일이 adr-014~018을 사용 중이며,
> **새 결정은 ADR-019부터 `.agent-os/adr/`에 파일로 추가**합니다(frontmatter: id·status·risk·related_specs).
> 이 파일은 기존 결정의 보관과 인덱스 역할을 유지합니다.

## ADR-001 단일 저장소 운영

- 결정: 문서, 백엔드, 프론트엔드를 한 저장소에서 관리합니다.
- 이유: 구조 파악이 쉽고 학습/개발 흐름을 한 번에 관리할 수 있습니다.

## ADR-002 문서 우선 개발

- 결정: 기능 구현 전 spec과 standards를 먼저 정리합니다.
- 이유: 구현과 문서가 어긋나면 유지보수가 어려워집니다.

## ADR-003 MVP는 단순 흐름 우선

- 결정: 복잡한 계정보다 이력서/JD 분석 핵심 흐름을 먼저 완성합니다.
- 이유: 가장 먼저 검증해야 할 가치는 분석 품질과 사용 흐름입니다.

## ADR-004 최종 배포는 분리 컨테이너형

- 결정: 프론트엔드와 백엔드는 각각 독립 컨테이너로 배포합니다.
- 이유: 역할 분리가 명확하고, 프론트 정적 자산과 백엔드 API를 독립적으로 교체하기 쉽습니다.
- 이유: 1.5차 MVP와 2차 MVP에서 업로드, 외부 AI 연동, 보안 설정이 늘어나도 배포 경계를 유지하기 좋습니다.
- 운영 기준: 로컬과 CI는 `compose.yaml`을 기준으로 통합 검증하고, 운영 환경에서도 reverse proxy 또는 ingress 뒤에서 `frontend`와 `backend`를 분리 유지합니다.

## ADR-005 실행 환경 중단과 run-state 기반 재개

- 결정: Codex 구현 루프는 실행 호스트의 상태에 의존합니다. Mac이 잠자기 또는 종료 상태가 되면 Codex 구현, `cron`·`launchd` 감지, Claude 리뷰 호출을 모두 중단합니다.
- 결정: 호스트가 깨어나거나 다시 시작되면 오케스트레이터가 영속적인 `run-state`를 읽고 마지막으로 저장된 단계부터 작업을 재개합니다.
- 결정: `run-state`에는 최소한 `run_id`, `spec_id`, 작업 브랜치, 현재 단계, 재시도 횟수, 마지막 완료 단계, 갱신 시각, 잠금 상태를 기록합니다.
- 결정: 실행 중단은 자동으로 spec을 새로 시작하거나 실패 처리하지 않습니다. spec의 생명주기 상태와 실행 중단 상태를 분리합니다.
- 결정: 이미 완료된 spec·단계·리뷰 시도는 멱등성 키와 상태 확인으로 다시 실행하지 않습니다. 외부 작업의 결과가 불확실한 경우에는 재실행 전에 GitHub 상태를 조회해 조정합니다.

### 이유

배터리 사용 환경에서는 Mac의 잠자기·종료를 피할 수 없습니다. 프로세스 메모리에만 상태를 두면 재부팅 후 같은 spec을 처음부터 구현하거나 PR·리뷰를 중복 실행할 수 있으므로, 실행 상태를 별도로 영속화해야 합니다.

### 영향

- Mac을 켜둔 동안에는 로컬 실행을 사용할 수 있습니다.
- 밤새 무인 실행이 필요하면 항상 켜진 VPS·홈서버·GitHub-hosted 실행 환경으로 오케스트레이터를 옮겨야 합니다.
- `run-state`의 구체적인 저장소와 복구 스크립트는 별도 오케스트레이터 구현 spec에서 정합니다.

## ADR-006 필수 테스트 의존성 중단 시 전체 spec 차단

- 결정: 현재 spec의 acceptance criteria 또는 test scenarios가 필수로 요구하는 DB·Redis·외부 서비스가 하나라도 준비되지 않으면 해당 spec 실행 전체를 중단합니다.
- 결정: 필수 의존성이 없는 상태에서는 구현 완료, 테스트 통과, 코드 리뷰, PR 생성·갱신, 다음 spec 진행을 허용하지 않습니다.
- 결정: 실행 상태에는 `status: blocked`, 차단 사유, 문제가 된 서비스 목록, 복구 후 재개할 단계(`resume_phase`)를 기록합니다.
- 결정: 의존성 복구 재시도는 코드 수정·리뷰 재시도 횟수와 분리합니다. 로컬 환경에서 Docker Desktop이나 Compose를 사용자 동의 없이 자동으로 다시 켜지 않습니다.

### 이유

필수 DB가 꺼진 상태에서 단위 테스트 일부만 실행하면 전체 검증이 끝난 것처럼 보이거나, 불완전한 결과로 PR과 다음 spec이 진행될 수 있습니다. 이를 막으려면 필수 의존성 하나의 중단을 현재 spec 전체의 차단 조건으로 취급해야 합니다.

### 재개 기준

의존성이 복구되면 오케스트레이터는 `run-state`를 읽고 저장된 `resume_phase`부터 재개합니다. 구현을 처음부터 다시 시작하지 않으며, 복구 전의 테스트 결과는 최종 통과 근거로 재사용하지 않습니다.

## DECISION-007 서비스 MVP 범위

- 결정: 서비스 MVP는 `OAuth 1개 공급자 로그인 → 이력서 저장 → JD 입력 또는 사람인 링크 수집 → AI 이력서 진단·JD 매칭 → 결과 저장 → 과거 결과 조회`의 완전한 사용자 흐름으로 정의합니다.
- 포함: 사용자 계정, 사용자 소유 데이터, PostgreSQL 기반 영속화 방향, 이력서·JD·분석 결과의 저장과 재조회.
- 제외: 추가 OAuth 공급자, JobKorea·RocketPunch 연동, ATS 분석, 모의면접, 결제, Redis 캐시·비동기 worker의 선제 도입.
- 기술 분리: PostgreSQL의 구체적인 스키마·마이그레이션·운영·복구 방식은 별도 technical ADR에서 확정한 뒤 구현합니다.

### 이유

기존 Phase 1~2.5는 기능 검증 중심의 MVP였고, 현재는 이력서와 분석 결과를 장기 보존하는 서비스 기반이 비어 있습니다. 서비스 MVP는 기능 목록이 아니라 사용자가 로그인부터 결과 재사용까지 한 번에 완료할 수 있는 제품 가치 단위로 정의합니다.

## DECISION-008 분석 이력 삭제 시 입력·결과 즉시 영구 삭제

- 결정: 사용자가 분석 이력을 삭제하면 해당 이력의 제출 당시 이력서 텍스트, JD, AI 분석 결과를 즉시 함께 영구 삭제합니다.
- 결정: PDF/DOCX 원본 파일은 애초에 텍스트 추출 후 보관하지 않습니다.
- 이유: 사용자의 삭제 의도를 입력 사본과 결과까지 일관되게 반영하고, 개인정보 최소 보관 원칙을 지킵니다.
- 영향: 삭제 후 해당 분석 이력은 조회·재시도할 수 없으며, 복구 기능은 제공하지 않습니다.

## DECISION-009 재시도는 새 분석 이력으로 저장

- 결정: 분석 재시도 결과는 기존 분석 이력을 덮어쓰지 않고 새 분석 이력으로 저장합니다.
- 결정: 재시도는 기존 분석 이력의 제출 당시 이력서 텍스트와 JD 사본을 그대로 사용합니다.
- 이유: 과거 결과와 재시도 결과를 비교할 수 있고, 제출 당시 결과의 불변성을 유지할 수 있습니다.
- 영향: 같은 입력으로 재시도해도 분석 이력이 추가됩니다.

## DECISION-010 1차 OAuth 공급자는 Google

- 결정: Service MVP의 단일 OAuth 공급자는 Google OAuth 2.0/OIDC로 시작합니다.
- 결정: MVP는 `openid`, `email`, `profile` 수준의 기본 식별 정보만 요청합니다.
- 이유: 기본 식별 스코프 중심으로 구현해 검수 부담을 낮추고, provider 토큰과 내부 사용자 식별을 분리할 수 있습니다.
- 영향: Kakao 등 추가 공급자는 MVP 범위에서 제외하며, 이후 별도 provider 경계로 추가합니다.
- 상세(Context·검토 대안·risk): [adr-015-oauth-provider](../adr/adr-015-oauth-provider.md) — 같은 결정의 기술 ADR. 내용 충돌 시 adr 파일이 정본.

## DECISION-011 이력서는 분석 이력의 입력 사본으로만 보존

- 결정: 별도의 이력서 라이브러리·작성 기능은 만들지 않고, 분석 요청 시 제출한 이력서 텍스트만 해당 분석 이력에 보존합니다.
- 결정: PDF/DOCX 원본 파일은 저장하지 않으며, 새 분석을 시작할 때 사용자가 기존 이력서를 다시 입력하거나 업로드합니다.
- 이유: JDSnack을 이력서 관리 도구가 아니라 기존 이력서 분석 도구로 한정하고, 보관 데이터를 최소화합니다.
- 영향: 재시도는 기존 분석 이력의 입력 사본을 사용하지만, 새 분석은 별도 입력으로 시작합니다.

## DECISION-012 JD 링크는 본문 스냅샷과 출처 URL을 저장

- 결정: 사람인 링크로 수집한 JD는 분석 당시 추출된 본문과 출처 URL을 함께 분석 이력에 저장합니다.
- 결정: 재시도·이력 조회에서는 저장된 본문 스냅샷을 사용하며 출처 URL을 다시 수집하지 않습니다.
- 이유: 당시 분석 내용을 재현하면서 원출처를 확인할 수 있고, 외부 공고 변경·삭제에 영향을 받지 않습니다.
- 영향: 저장된 URL은 출처 표시용 메타데이터이며 최신 JD 동기화 기능은 MVP 범위에 포함하지 않습니다.

## DECISION-013 AI 분석 실패도 실패 상태 이력으로 저장

- 결정: AI 분석이 실패해도 제출 당시 이력서 텍스트와 JD를 포함한 분석 이력을 `FAILED` 상태로 저장합니다.
- 결정: 실패 이력의 재시도는 기존 이력을 덮어쓰지 않고 새 분석 이력을 생성합니다.
- 이유: 사용자가 입력을 다시 제출하지 않고 재시도할 수 있으며, 분석 요청의 성공·실패 상태를 명확히 확인할 수 있습니다.
- 영향: 실패 이력에도 사용자 입력 사본이 보존되므로 일반 분석 이력과 동일한 삭제 정책을 적용합니다.
