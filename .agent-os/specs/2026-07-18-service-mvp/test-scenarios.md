# Service MVP 테스트 시나리오

## 인증

- **TC-01 Google 로그인 성공**: fake OAuth provider callback으로 사용자 식별자를 생성하고 내부 사용자 세션을 발급한다. provider secret/token이 응답과 프론트 로그에 포함되지 않는지 확인한다.
- **TC-01a 인증 UI 게이트**: 세션이 없으면 공개 홈 화면과 로그인 CTA를 표시하고, 분석 입력·JD 수집·분석 실행 입력과 버튼은 잠겨 있는지 확인한다. CTA 클릭 시 Google 로그인 모달을 열고 닫을 수 있으며, 로그인 전 보호 API 호출이 발생하지 않아야 한다. 인증 세션에서는 보호 기능을 사용할 수 있다.
- **TC-02 인증 없는 보호 API**: 세션 없는 기존 분석 API와 분석 생성·목록·상세·재시도·삭제 요청이 controller/service를 실행하지 않고 `401 AUTHENTICATION_REQUIRED`로 종료되는지 확인한다.

## 입력과 분석

- **TC-03 텍스트 이력서 + 텍스트 JD 성공**: 정규화된 텍스트 입력으로 기존 진단·매칭 fake를 호출하고 `SUCCEEDED` 이력에 입력과 결과가 저장되는지 확인한다.
- **TC-04 PDF/DOCX 입력**: 기존 추출 경계를 통해 텍스트를 얻은 뒤 분석을 실행하고 원본 파일 바이트가 저장소에 남지 않는지 확인한다.
- **TC-05 사람인 JD 링크 성공**: 기존 JD fetch fake가 본문과 URL을 반환하면 본문 스냅샷과 출처 URL을 저장한다.
- **TC-06 JD 수집 실패**: JD fetch 실패 시 분석을 실행하지 않고 공통 외부 입력 오류를 반환한다.
- **TC-07 AI 실패 이력**: AI provider가 예외 또는 검증 실패를 반환하면 입력 스냅샷을 가진 `FAILED` 이력이 저장되고 사용자에게 재시도 ID가 제공되는지 확인한다.

## 조회와 소유권

- **TC-08 목록 조회**: 사용자 A가 자신의 이력만 시간순으로 조회하고 사용자 B의 이력은 포함되지 않는지 확인한다.
- **TC-09 타인 이력 접근**: 사용자 A가 사용자 B의 ID로 상세·재시도·삭제를 요청할 때 존재 여부가 드러나지 않는 공통 리소스 오류를 확인한다.
- **TC-10 상세 조회**: 성공 이력에는 입력 스냅샷·JD·결과가, 실패 이력에는 상태·실패 사유·입력 스냅샷이 반환되는지 확인한다.

## 재시도와 삭제

- **TC-11 성공 이력 재시도**: 저장된 입력으로 새 분석을 실행하고 새 ID를 발급하며 기존 이력은 변하지 않는지 확인한다.
- **TC-12 실패 이력 재시도**: 실패 이력의 입력을 재사용해 새 이력을 만들고 기존 실패 이력은 보존되는지 확인한다.
- **TC-13 URL 재수집 금지**: 링크 기반 이력 재시도에서 JD fetch adapter가 호출되지 않고 저장된 본문이 사용되는지 확인한다.
- **TC-14 삭제 연쇄**: 이력 삭제 후 입력 스냅샷·JD·결과와 이력 레코드가 함께 사라지고 재조회·재시도가 실패하는지 확인한다.
- **TC-15 반복 삭제와 멱등성**: 이미 삭제된 이력의 재삭제가 안전한 공통 리소스 오류로 종료되고 다른 이력은 영향을 받지 않는지 확인한다.

## 회귀

- **TC-16 기존 AI 진단·JD 매칭 회귀**: 기존 fixture/stub 계약과 프론트 입력·결과 흐름이 유지되는지 확인한다.
- **TC-17 계층·비밀값 회귀**: Controller가 비즈니스 로직을 갖지 않고, 프론트 컴포넌트가 직접 API를 호출하지 않으며, 비밀값이 클라이언트 번들에 포함되지 않는지 확인한다.
