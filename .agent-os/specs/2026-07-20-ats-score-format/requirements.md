# ATS 점수·포맷 진단 요구사항

## 목표

결과 화면에 남은 마지막 "준비중" 분석 옵션인 **ATS 분석**을 실동작으로 해금합니다. 로그인한 사용자가 이력서·JD를 제출하면 이력서가 기업의 자동 파싱·필터링 시스템(ATS)을 얼마나 안전하게 통과하는지를 **파싱 안전성 + 구조 진단 + JD 대비 키워드 최적화** 세 관점("기계가 찾는가")으로 진단하고, 종합 ATS 점수와 함께 제공합니다.

"ATS 분석"의 정의 정본은 [용어집](../../product/glossary.md)이며, 결정 근거는 [2026-07-16 spec backlog grill decisions](../../archive/research/2026-07-16-spec-backlog-grill-decisions.md)(라인 118-122)에 있습니다.

## 범위

### 포함

- 프론트 분석 옵션 `ats`를 `enabled: true`로 해금하고 결과 화면의 `ComingSoonPanel`을 실제 ATS 패널로 교체
- 신규 전용 엔드포인트 `POST /api/ats/preview` (매칭·문장 첨삭과 동일한 요청 계약·검증·오류 체계 재사용)
- 파싱 안전성 진단: 표·이미지 placeholder·비표준 섹션 등 기계 파싱 위험 요소 감지
- 구조 진단: 표준 이력서 섹션 유무와 순서 진단
- JD 대비 키워드 최적화: JD가 요구하는 키워드가 이력서에서 기계적으로 발견되는지 진단
- 세 진단을 종합한 ATS 점수와 요약
- `stub`/`fixture`/`ai-local` 세 모드 지원과 결과 내보내기(마크다운) ATS 섹션

### 제외

- 새 `ErrorCode`, `ApiResponse<T>` 래퍼 변경, 매칭·문장·인터뷰 등 기존 엔드포인트의 요청/응답 계약 변경
- 원본 PDF/DOCX 바이너리 레이아웃 분석(제출 이력서는 이미 추출된 텍스트를 대상으로 한다. 기존 텍스트 추출 경계를 그대로 사용)
- 형태소 분석기·임베딩 등 키워드 추출 알고리즘 고도화(기존 토큰 기반 추출 위에서 진단 구조화만)
- 분석 이력(Analysis History) 저장 스키마·저장 계약 변경. 본 spec은 preview 계층만 다룬다
- 다른 "준비중" 옵션은 이미 없으므로 추가 해금 대상 없음(ATS가 마지막 잠금 옵션)

## 요구사항

### REQ-01 ATS 분석 옵션 해금

프론트 `ANALYSIS_OPTIONS`의 `ats` 항목을 `enabled: true`로 바꾸고, 결과 화면에서 `ats` 선택 시 `ComingSoonPanel`("준비중") 대신 실제 ATS 진단 패널을 렌더링합니다. `ats`는 옵션 목록에서 선택 가능해야 하며 "준비중" 태그를 갖지 않습니다.

### REQ-02 ATS 분석 엔드포인트와 계약

신규 전용 엔드포인트 `POST /api/ats/preview`를 추가합니다. 요청 형태는 매칭(`MatchPreviewRequest`)·문장 첨삭(`SentencePreviewRequest`)과 동일한 `resumeSource`/`jdText`/`jdUrl` 구조이며, 동일한 입력 검증(이력서/JD 길이 50~10,000자, `jdUrl` 검증)과 기존 `ErrorCode` 체계, `ApiResponse<T>` 래퍼를 재사용합니다. 새 `ErrorCode`는 추가하지 않습니다.

### REQ-03 파싱 안전성 진단

서버는 제출된 이력서 텍스트에서 기계 파싱을 방해하는 위험 요소(표 구조, 이미지 placeholder, 비표준·해독 불가 섹션 제목, 다단 레이아웃 흔적 등)를 감지해 경고 목록으로 제공합니다. 위험 요소가 없으면 빈 목록을 반환합니다.

### REQ-04 구조 진단

서버는 표준 이력서 섹션(예: 경력, 학력, 기술/스킬, 프로젝트 등)의 존재 여부와 순서를 진단하고, 감지된 섹션·누락된 섹션·순서 경고를 제공합니다.

### REQ-05 JD 대비 키워드 최적화

서버는 JD가 요구하는 키워드가 이력서에서 기계적으로 발견되는지("ATS가 찾는가") 관점으로 진단해 반영된 키워드와 누락된 키워드를 제공합니다. 이는 JD 적합도(`JD Match`)의 인간 관점 분석과 별개로, 기계 검색 가능성 관점을 다룹니다.

### REQ-06 종합 ATS 점수

서버는 파싱 안전성·구조·키워드 세 진단을 종합해 0~100의 ATS 통과 가능성 점수와 요약 문구를 제공합니다.

### REQ-07 모드별 결정성과 관대 처리

`stub`/`fixture` 모드는 외부 호출 없이 결정적 결과(동일 입력 → 동일 출력)를 반환합니다. `ai-local` 모드는 Gemini를 실호출합니다. 모든 응답 필드는 항상 직렬화되며(목록은 빈 배열, 점수는 기본값) null이 아닙니다. Gemini 응답의 진단 필드는 관대하게 처리합니다(누락/비배열이면 기본값으로 채우고 성공 유지). 기존 필수 검증·`GEMINI_API_*` 정책은 변경하지 않습니다.

### REQ-08 계층과 계약 경계

ATS 진단은 신규 패키지 `com.jdsnack.ats`의 `Controller -> Service -> Provider(External API)` 경계로 구현하고 Entity를 응답으로 직접 반환하지 않습니다. 프론트 컴포넌트는 직접 API를 호출하지 않고 `services/` 계층을 통해서만 호출합니다. 비밀 키 등 민감정보는 서버에서만 처리합니다. 계층·경계 규칙 정본은 [coding-standards](../../standards/coding-standards.md)와 [api 표준](../../standards/api.md)입니다.

### REQ-09 결과 표시와 내보내기

결과 화면의 ATS 패널은 종합 점수·요약과 파싱 안전성·구조·키워드 세 섹션을 표시하며, 로딩·에러·빈 상태를 기존 `AnalysisPanel` 패턴과 동일하게 처리합니다. 결과 내보내기 마크다운(`buildResultMarkdown`)에 `ats` 선택 시 ATS 섹션을 추가합니다.
