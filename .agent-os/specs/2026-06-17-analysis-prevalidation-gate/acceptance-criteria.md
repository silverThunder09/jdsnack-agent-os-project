# 분석 실행 전 통합 검증 게이트 수용 기준

## `AC-01` 통합 검증 결과 산출

- 이력서·JD·분석 항목을 한 번에 평가해, 전체 통과 여부와 항목별 실패 사유를 산출한다.
- 텍스트 모드 이력서가 50자 미만이거나 빈 값이면 미통과로 판정한다.
- 파일 모드 이력서는 "지원 형식 파일이 선택됨"이면 프론트 게이트 기준 통과로 판정한다(추출 본문 길이는 백엔드가 재검증).
- JD가 빈 값이거나 50자 미만이거나 10,000자 초과면 미통과로 판정한다.
- 분석 항목이 0개면 미통과로 판정한다.

## `AC-02` 게이팅: 버튼 비활성 + 인라인 사유

- 검증 미통과 상태에서는 분석 시작 버튼이 비활성이다.
- 미통과 사유(예: "JD를 50자 이상 입력해 주세요", "분석 항목을 1개 이상 선택해 주세요")가 인라인으로 표시된다.
- 모든 항목이 통과하면 버튼이 활성화된다.

## `AC-03` 통과 전 호출·업로드 차단

- 검증 미통과 상태에서 분석을 시도해도 `POST /api/diagnose`, `POST /api/diagnose/file`, `POST /api/match/preview` 요청이 발생하지 않는다.
- 짧은 JD(50자 미만) + 정상 파일 조합에서, 파일 업로드/추출 요청(`/api/diagnose/file`)이 먼저 발생하지 않는다(검증이 호출보다 앞선다).

## `AC-04` 프론트·백엔드 동일 기준·방어적 재검증

- 프론트와 백엔드의 최소/최대 길이 기준이 동일하다(이력서 50/10,000, JD 50/10,000).
- 프론트 게이트를 우회한 요청이 백엔드에 도달하면, 진단/매칭이 동일 기준으로 재검증해 `400`과 해당 `ErrorCode`(`EMPTY_RESUME`, `TEXT_TOO_SHORT`, `TEXT_TOO_LONG`, `EMPTY_JD`, `JD_TEXT_TOO_SHORT`, `JD_TEXT_TOO_LONG`)를 반환한다.
- 파일 추출 본문이 50자 미만이면 백엔드가 `TEXT_TOO_SHORT`로 막는다.
- 응답은 기존 `ApiResponse<T>` 래퍼와 에러 코드 체계를 그대로 따른다(새 코드·스키마 추가 없음).

## `AC-05` 기존 흐름·검증·접근성 보존

- 게이트 통과 후의 진단→매칭 실행 순서와 결과 표시가 기존과 동일하다.
- `stub`/`fixture`/`ai-local` 모드 동작, 기존 에러 처리(키 누락·Gemini 실패 등)가 유지된다.
- 비활성 버튼·인라인 사유는 명확한 label/role과 `aria-disabled`(또는 동등) 접근성 기준을 따르고, 좁은 폭 레이아웃을 깨지 않는다.
