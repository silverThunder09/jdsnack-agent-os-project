## 3. 리스크 및 대응 계획

### 3.1 Gemini API 관련 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|-----------|--------|-----------|
| API 응답 지연 (30초 초과) | 중 | 높음 | 타임아웃 설정 + 사용자에게 재시도 안내 메시지 표시 |
| API 할당량 초과 | 낮음 | 높음 | 개발 중 호출 횟수 절약, 캐시 활용 검토 |
| API 응답 형식 불일치 | 중 | 중 | JSON 파싱 실패 시 기본 에러 메시지 반환, 로그 기록 |
| API 키 노출 | 낮음 | 매우 높음 | 환경 변수로만 관리, `.gitignore`에 설정 파일 추가 |
| Gemini 서비스 자체 장애 | 낮음 | 높음 | 사용자에게 일시적 장애 안내 + 재시도 버튼 제공 |

**최악의 상황 대응:** Gemini API가 완전히 사용 불가할 경우, 하드코딩된 Mock 응답으로 데모를 진행할 수 있도록 `MockGeminiService`를 미리 준비한다.

### 3.2 AWS 배포 관련 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|-----------|--------|-----------|
| EC2 인스턴스 설정 실패 | 중 | 높음 | Elastic Beanstalk으로 전환 (더 간편한 배포) |
| 포트/보안그룹 설정 오류 | 중 | 중 | 체크리스트 기반으로 하나씩 확인 |
| Java 17 런타임 미설치 | 낮음 | 중 | Amazon Corretto 17 설치 스크립트 사전 준비 |
| 메모리 부족 (t2.micro) | 중 | 중 | JVM 힙 사이즈 제한 (`-Xmx256m`) |
| AWS 계정/비용 문제 | 낮음 | 높음 | 프리 티어 한도 사전 확인 |

**백업 계획:** AWS 배포가 완전히 실패할 경우, 로컬 환경에서 ngrok을 사용하여 임시 퍼블릭 URL을 생성하고 데모를 진행한다.
