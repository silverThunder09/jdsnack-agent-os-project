# 프론트 SaaS 셸 재설계 요구사항

대시보드 화면(입력 레일 | 결과 3패널, 분석 버튼 3개 분절)의 흐름이 약하다는 판단으로, 제공된 목업(SaaS형 셸)을 기준으로 프론트의 비주얼·UX를 전면 재설계한다. **frontend 한정이며 백엔드 API·기능 동작·계약은 바꾸지 않는다.** 목업의 풍성한 기능 중 백엔드가 없는 것(ATS·키워드 구조화·문장 첨삭·인증·내역·요금제 등)은 이번 범위에서 구현하지 않고, **비주얼/UX 셸만** 채택해 기존 기능(진단/매칭/면접)에 매핑한다.

## 문제 정의 (현재 화면)

- 분석 실행 버튼이 진단/매칭/면접 3개로 분절돼 하나의 분석 여정으로 읽히지 않는다.
- 입력·결과 의존관계(매칭=이력서+JD)가 화면에 드러나지 않는다.
- 첫 진입 시 빈 결과 패널만 보여 무엇부터 할지 모호하다.
- 제품 정체성을 주는 셸(내비게이션·히어로)이 없어 단일 화면이 평이하다.

## `REQ-01` SaaS 셸 레이아웃

- 좌측 사이드바 + 상단 바 + 본문 콘텐츠 영역으로 구성된 셸을 도입한다.
- 본문은 히어로 → 통합 입력 → 결과 영역 순의 단일 화면 흐름을 가진다.
- 브랜드 식별은 JDSnack을 유지한다(제품명 변경 없음).

## `REQ-02` 사이드바 내비게이션

- 목업의 전체 메뉴(홈·분석 내역·이력서 관리·템플릿·키워드 사전·맞춤 첨삭·요금제)와 모의 면접을 노출한다.
- 실제 동작하는 항목(홈, 모의 면접)만 활성화하고, 백엔드가 없는 항목은 잠금(비활성, 클릭 불가)으로 표시한다.
- 인증이 없으므로 계정/프로필·플랜 영역은 표시하지 않는다.

## `REQ-03` 통합 분석 진입

- 채용 공고(JD 링크/붙여넣기)와 이력서(텍스트/PDF/DOCX)를 한 입력 영역에 모은다.
- 단일 `분석 시작`이 이력서 진단과 JD 매칭을 함께 실행한다.
- 입력값은 실행 후에도 보존되어 재실행할 수 있다.

## `REQ-04` 결과 표시

- 결과 요약 카드는 데이터가 존재하는 지표(이력서 진단 점수, JD 적합도=매칭 점수)만 노출한다.
- 진단(강점/개선)과 매칭(강점/gap/제안) 상세를 프리뷰 영역에 표시한다.
- 결과 생성 전 빈/안내, 생성 중 로딩, 실패 시 에러+입력 보존을 각 영역에서 처리한다.

## `REQ-05` 모의 면접 목적지 유지

- 모의 면접 질문 생성을 별도 목적지(라우트 또는 뷰)로 유지한다.
- 대상 직무(선택적으로 JD/이력서 맥락) 입력 후 질문(카테고리·키포인트)·전략·요약을 표시한다.

## `REQ-06` 기존 기능·계약·검증 보존

- `POST /api/diagnose`, `/diagnose/file`, `/api/jd/fetch`, `/api/match/preview`, `/api/interview/preview` 계약을 변경하지 않는다.
- 입력 검증, 에러 처리, `stub`/`fixture`/`ai-local` 모드 동작을 그대로 유지한다.
- resume source(텍스트/파일 추출 `sourceText`)를 진단·매칭·면접에 동일하게 사용한다.

## `REQ-07` 반응형·접근성

- 데스크톱은 사이드바|본문, 본문 내 입력|결과의 넓은 레이아웃을 사용한다.
- 좁은 폭에서는 세로 스택으로 전환하고 사이드바는 접힘/축소 형태로 처리한다.
- label/role/오류 알림(`role="alert"` 등)/포커스 흐름을 유지한다.
